# ContriSplit

팀원의 프로젝트 기여도를 CBT로 기록하고, 확정된 지분에 따라 프로젝트 보상금을 분배하는 스마트 컨트랙트 데모 프로젝트입니다.

ContriSplit은 팀 프로젝트에서 기여도 산정과 보상금 분배 기준이 모호해질 수 있는 문제를 해결하기 위해 만든 데모입니다.

관리자가 팀원의 기여도를 CBT로 기록하고, 프로젝트 종료 후 지분을 확정하면 RewardVault가 CBT 보유 비율에 따라 각 팀원에게 보상금을 계산합니다.

각 팀원은 자신의 계정으로 직접 보상금을 출금할 수 있습니다.

## 프로젝트 개요

이 프로젝트는 로컬 Hardhat 환경에서 동작하는 스마트 컨트랙트 기반 데모입니다.

- 팀원의 기여도를 토큰 수량으로 기록합니다.
- 프로젝트 종료 후 지분을 확정합니다.
- 보상금은 CBT 비율에 따라 계산됩니다.
- 각 팀원은 자신의 보상금을 직접 출금합니다.

## 문제 정의

팀 프로젝트나 공동 외주 작업에서는 구성원마다 기여한 정도가 다를 수 있습니다.

하지만 기여도를 명확한 기준으로 기록하지 않으면 보상금을 나눌 때 다음 문제가 발생할 수 있습니다.

- 기여도 기준이 명확하지 않습니다.
- 보상 비율을 사람이 직접 계산해야 합니다.
- 계산 과정에서 실수가 발생할 수 있습니다.
- 누가 얼마를 받아야 하는지에 대한 분쟁이 생길 수 있습니다.
- 지급 과정이 특정 관리자에게 지나치게 의존할 수 있습니다.

## 해결 방식

ContriSplit은 팀원의 기여도를 CBT 수량으로 기록합니다.

CBT는 거래용 토큰이 아니라 프로젝트 기여 지분을 나타내는 기록용 토큰입니다. 따라서 사용자 간 전송은 차단됩니다.

프로젝트 종료 후 관리자가 지분을 확정하면 CBT 발행과 회수가 차단됩니다.

이후 RewardVault에 보상금이 입금되면 각 팀원의 CBT 비율에 따라 보상금이 계산됩니다.

각 팀원은 claimReward()를 직접 호출해 자신의 몫만 출금할 수 있습니다.

## 핵심 시나리오

다음 예시를 기준으로 동작합니다.

| 팀원 | CBT | 기여 지분 | 보상금 |
|---|---:|---:|---:|
| A | 50 CBT | 50% | 0.5 ETH |
| B | 30 CBT | 30% | 0.3 ETH |
| C | 20 CBT | 20% | 0.2 ETH |
| 합계 | 100 CBT | 100% | 1 ETH |

## 주요 기능

### CBT 기여도 기록

- 관리자만 CBT를 발행할 수 있습니다.
- 잘못 지급된 CBT는 관리자가 회수할 수 있습니다.
- CBT는 정수 단위로만 사용합니다.
- 사용자 간 CBT 전송은 차단됩니다.

### 프로젝트 지분 확정

- CBT 지급 완료 후 관리자가 지분을 확정합니다.
- 확정 후에는 추가 발행과 회수가 불가능합니다.
- CBT가 발행되지 않은 상태에서는 지분을 확정할 수 없습니다.

### 프로젝트 보상금 분배

- 프로젝트 지분 확정 후에만 보상금을 입금할 수 있습니다.
- 관리자만 RewardVault에 보상금을 입금할 수 있습니다.
- CBT 보유 비율에 따라 팀원별 보상금이 계산됩니다.
- 각 팀원이 자신의 보상금을 직접 출금합니다.
- 동일한 보상금의 중복 출금은 차단됩니다.

### 보안 처리

- 관리자 권한은 OpenZeppelin Ownable로 관리합니다.
- 출금 함수에는 ReentrancyGuard를 적용합니다.
- ETH 전송 전에 출금 상태를 먼저 갱신합니다.
- ETH 전송 실패 시 전체 트랜잭션을 되돌립니다.

## 시스템 구성

```text
관리자
  │
  ├─ 팀원별 CBT 지급
  ▼
ContributionToken
  │
  ├─ CBT 잔액
  ├─ 전체 발행량
  └─ 지분 확정 여부
  ▼
RewardVault
  │
  ├─ 보상금 입금
```

## ContributionToken

ContributionToken.sol은 팀원의 기여도를 CBT 수량으로 기록합니다.

| 함수 | 설명 |
|---|---|
| mint() | 관리자가 팀원에게 CBT를 지급합니다. |
| burnFrom() | 관리자가 잘못 지급된 CBT를 회수합니다. |
| finalize() | 프로젝트 기여 지분을 최종 확정합니다. |
| decimals() | 토큰의 소수점 자릿수를 0으로 설정합니다. |
| balanceOf() | 팀원의 CBT 보유량을 조회합니다. |
| totalSupply() | 전체 CBT 발행량을 조회합니다. |
| finalized() | 프로젝트 지분 확정 여부를 조회합니다. |

CBT는 거래용 토큰이 아니므로 사용자 간 transfer()와 transferFrom()은 차단됩니다.

## RewardVault

RewardVault.sol은 프로젝트 보상금을 보관하고 CBT 비율에 따라 각 팀원의 보상금을 계산합니다.

| 함수 | 설명 |
|---|---|
| depositReward() | 관리자가 테스트용 ETH 보상금을 입금합니다. |
| totalEntitlement() | 팀원이 받을 수 있는 전체 누적 보상금을 계산합니다. |
| claimable() | 현재 추가로 출금 가능한 금액을 계산합니다. |
| claimReward() | 호출자가 자신의 보상금을 직접 출금합니다. |
| vaultBalance() | RewardVault에 남아 있는 ETH를 조회합니다. |

totalRewardDeposited는 지금까지 입금된 누적 보상금이고, vaultBalance()는 팀원들의 출금 이후 현재 남아 있는 Vault 잔액입니다.

## 보상금 계산 방식

```text
팀원의 전체 보상 권리
= 누적 보상금 × 팀원의 CBT ÷ 전체 CBT
```

현재 출금 가능 금액은 다음과 같습니다.

```text
현재 출금 가능 금액
= 전체 보상 권리 - 이미 출금한 금액
```

예를 들어 A가 50 CBT를 보유하고 전체 CBT가 100이며 보상금이 1 ETH인 경우 다음과 같습니다.

```text
1 ETH × 50 ÷ 100 = 0.5 ETH
```

## 보안 및 검증 요소

### 관리자 권한

Ownable을 이용해 CBT 발행, 회수, 지분 확정 및 보상금 입금 권한을 제한합니다.

### 사용자 간 전송 차단

CBT는 기여도 기록용 토큰이므로 사용자 주소 간 토큰 이동을 차단합니다.

### 지분 확정 이후 변경 차단

프로젝트 지분이 확정된 이후에는 CBT를 추가로 발행하거나 회수할 수 없습니다.

### 중복 출금 방지

팀원별 누적 출금액을 기록해 이미 수령한 보상금을 다시 출금하지 못하게 합니다.

### 재진입 공격 방지

claimReward()에 nonReentrant를 적용하고, ETH 전송 전에 출금 상태를 먼저 갱신합니다.

### ETH 전송 실패 처리

ETH 전송이 실패하면 전체 트랜잭션을 되돌립니다.

## 프로젝트 한계점

- 프론트엔드는 스마트 컨트랙트와 직접 연결되지 않습니다.
- Hardhat 로컬 네트워크에서만 검증했습니다.
- MetaMask와 같은 실제 지갑 연결을 지원하지 않습니다.
- 실제 테스트넷 또는 메인넷에 배포하지 않았습니다.
- 팀원과 CBT 수량은 데모 시나리오에 맞춰 고정되어 있습니다.
- 여러 프로젝트를 하나의 컨트랙트에서 동시에 관리하지 않습니다.
- 보상금은 테스트용 ETH만 사용합니다.
- 정수 나눗셈으로 인해 임의의 비율에서는 소량의 잔여 wei가 남을 수 있습니다.
- 관리자가 CBT를 발행하고 회수하므로 완전한 탈중앙화 구조는 아닙니다.

관리자가 존재하므로 완전한 탈중앙화는 아닙니다. 그러나 프로젝트 기여도는 자동으로 판단되는 값이 아니므로 초기 평가와 수정 권한이 필요합니다. 지분 확정 이후에는 관리자의 변경 권한도 제한됩니다.

## 확장 방향

- 프로젝트별로 별도의 기여도 지분을 관리하는 구조
- 여러 프로젝트를 동시에 관리하는 팩토리 컨트랙트
- 팀원 투표를 통한 CBT 지급 승인
- 다중 서명 방식의 관리자 권한
- 실제 ERC-20 스테이블코인을 이용한 보상금 지급
- MetaMask 또는 지갑 연동
- 스마트 컨트랙트 이벤트 기반 프론트엔드 상태 조회
- 테스트넷 배포
- 기여도 평가 기록과 근거 저장
- 남은 잔여 wei 처리 방식 추가

## 발표 시 주의사항

- 프론트 버튼이 실제 스마트 컨트랙트를 호출한다고 설명하지 않습니다.
- 실제 컨트랙트 동작은 Hardhat 테스트와 demo.ts에서 검증했다고 설명합니다.
- CBT는 돈이나 거래용 코인이 아니라 기여 지분 기록용 토큰이라고 설명합니다.
- 보상금은 관리자가 팀원에게 직접 송금하는 것이 아니라 각 팀원이 직접 출금한다고 설명합니다.
- 50:30:20은 데모를 위한 예시 기여도입니다.
- 블록체인이 기여도를 자동으로 판단하는 것은 아니라고 설명합니다.
- 관리자가 초기 기여도를 기록하고 프로젝트 종료 후 지분을 확정한다고 설명합니다.
